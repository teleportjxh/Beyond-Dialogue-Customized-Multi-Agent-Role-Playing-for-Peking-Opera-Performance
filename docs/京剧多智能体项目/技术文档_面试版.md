# 京剧多智能体角色扮演项目（Beyond Dialogue）面试技术文档

> Beyond Dialogue：面向京剧表演的定制化多智能体剧本生成系统。核心是让多个专业 AI 智能体协作，基于真实历史演出数据，为特定京剧角色创作个性化剧本。

---

## 0. 先认识核心概念——从零理解

> 在讲项目细节之前，先搞清楚三个关键问题：为什么传统文本生成不够用、RAG 是什么、多智能体协作是什么。

### 0.1 为什么不能直接让 ChatGPT 写京剧剧本？

#### 通用 LLM 的"无个性"问题

对 GPT-4 说"帮我写一段诸葛亮在空城计中的唱词"，它能生成一段京剧风格的文本。但有几个根本性问题：

**问题1：没有角色个性**

不同演员版本的诸葛亮唱法、念白风格截然不同。李少春版、叶盛兰版、于魁智版的诸葛亮各有特色。通用 LLM 只知道"诸葛亮是丞相"，不知道"特定角色在特定行当的表演规范"。

**问题2：不懂京剧规范**

京剧有严格的格式规范：
- 西皮二黄是两种调式，不能混用
- 唱词有固定的板式（原板、慢板、流水板、散板）
- 念白分韵白（文言）和散白（口语），场合不同用法不同
- 动作有专用术语（亮相、圆场、水袖等），不能乱描述

**问题3：没有历史数据支撑**

面对同一剧情，要生成"符合这个角色历史表演风格的剧本"，必须能检索到该角色的真实历史唱段作为参照。

**Beyond Dialogue 解决的正是这三个问题**：通过 RAG 提供历史数据支撑，通过角色档案（Profile）固化角色个性，通过多智能体各司其职确保京剧规范。

---

### 0.2 什么是 RAG（检索增强生成）？

#### 从"记忆"到"查资料"

LLM 的知识来自训练数据，训练结束后就固定了——它不会自动了解"2023年某演员的最新演出风格"。

**RAG（Retrieval-Augmented Generation）** 的核心思路：
1. 提问时，先去知识库里搜索相关资料
2. 把搜索结果拼接到提示词里
3. 让 LLM 基于这些资料回答

```
不用RAG：LLM凭记忆回答 → 可能编造、缺乏特异性
用 RAG：LLM + 检索到的真实资料 → 有据可查、个性化
```

**在本项目中**：
- 知识库 = 56个角色的历史剧本（增强后的结构化版本）
- 检索 = 当写到某段剧情时，搜索"历史上这个角色在类似场景下唱了什么"
- 增强 = 把检索结果注入生成提示词，让 LLM 模仿历史风格

---

### 0.3 什么是多智能体系统？为什么一个 Agent 不够？

#### 分工协作 vs 独自完成

想象一下舞台剧的创作过程：
- **导演**负责整体把控，决定主题、节奏、风格
- **编剧**负责故事情节，写剧情大纲
- **演员**负责台词表演，体现角色个性
- **服装设计师**负责角色装扮，设计符合身份的行头
- **舞台设计师**负责场景，安排布景、灯光、道具

如果只用一个 AI 同时做所有事情，提示词会极其复杂，输出也会各方面都不够精细。

**多智能体（Multi-Agent）** 的优势：
- 每个 Agent 专注于自己的职责域，提示词更聚焦
- Agent 之间可以相互参考输出（演员参考编剧的大纲，服装设计参考演员的角色设定）
- 迭代优化更容易（单独改进某个 Agent，不影响整体）

## 1. 项目概述

### 核心定位

**问题**：传统京剧角色扮演生成系统无法捕捉特定角色的历史表演个性，生成的内容缺乏京剧专业规范性。

**解决方案**：构建以真实历史演出数据为基础的 RAG 增强多智能体系统，五个专业 Agent 协作生成完整的定制化京剧剧本（含唱腔、念白、身段、服装、场景设定）。

**类比**：相当于把真实京剧创作团队（导演+编剧+演员+服装+舞台）数字化，并以历史演出档案为知识库。

### 系统特性

| 特性 | 描述 |
|------|------|
| **角色定制化** | 基于角色历史剧本数据构建专属档案（Profile），生成结果忠于该角色风格 |
| **RAG 语义检索** | FAISS 向量索引，按语义相似度检索历史表演片段 |
| **多智能体协作** | 5个专业化 Agent，各司其职，输出相互依赖和参照 |
| **京剧规范输出** | 严格遵循西皮/二黄、板式、念白、身段等专业规范 |
| **全要素生成** | 不只有剧本，还包括大纲、场景设定 JSON、装扮设计 JSON、评估报告 |
| **多角色支持** | 已支持诸葛亮（21部）、孙悟空（13部）、赵匡胤（22部） |

## 3. 核心技术实现

### 3.1 数据提取与结构化

**原始数据**：京剧剧本 PDF/TXT，格式不统一，混杂唱词、念白、身段说明。

**提取流程**：

```python
# extractor.py 核心逻辑
def extract_script(file_path: str) -> ScriptData:
    # 1. 读取原始文本
    raw_text = read_file(file_path)
    
    # 2. 预处理：去除页码、版权信息等干扰
    cleaned = clean_text(raw_text)
    
    # 3. 调用 LLM 结构化提取
    prompt = f"""
    请从以下京剧剧本中提取结构化信息：
    - 角色名及行当
    - 每段唱腔的类型（西皮/二黄）、板式、完整唱词
    - 念白类型（韵白/散白）和内容
    - 身段动作描述
    
    剧本内容：{cleaned[:5000]}
    
    输出 JSON 格式...
    """
    return llm_client.extract(prompt)
```

**输出结构**（`character_data/诸葛亮/data.json` 节选）：

### 3.2 RAG 向量索引构建

**为什么用 FAISS**：

京剧剧本检索需要**语义相似性**，不是关键词匹配。例如"诸葛亮遭遇危机时的应对"应该能检索到《空城计》，即使检索词和剧本原文没有完全重叠的词。

```python
# vector_processor.py 核心
class VectorProcessor:
    def build_index(self, enhanced_scripts_dir: str):
        documents = []
        embeddings = []
        
        for script_file in glob(f"{enhanced_scripts_dir}/**/*.txt"):
            # 1. 读取增强剧本，按段落切片
            chunks = self.split_into_chunks(script_file, chunk_size=512, overlap=64)
            
            for chunk in chunks:
                # 2. 调用 embedding API 向量化
                vector = self.embed(chunk.text)  # 1536维向量
                
                documents.append({
                    "text": chunk.text,
                    "character": chunk.character,
                    "play": chunk.play,
                    "chunk_type": chunk.type  # singing/dialogue/action
                })
                embeddings.append(vector)
        
        # 3. 构建 FAISS 索引
        self.index = faiss.IndexFlatL2(1536)  # L2距离
        self.index.add(np.array(embeddings, dtype=np.float32))
        
        # 4. 持久化
        faiss.write_index(self.index, "vector_index/faiss.index")
        json.dump(documents, open("vector_index/documents.json", "w"))
```

**检索时**：
```python
def retrieve(self, query: str, character: str, top_k: int = 5):
    query_vec = self.embed(query)
    distances, indices = self.index.search(
        np.array([query_vec], dtype=np.float32), top_k * 3
    )
    # 过滤：只保留目标角色的片段
    results = [
        self.documents[i] for i in indices[0]
        if self.documents[i]['character'] == character
    ]
    return results[:top_k]
```

### 3.3 多智能体协作机制

**Agent 基类设计**：

```python
# agent_base.py
class AgentBase:
    def __init__(self, character_profile: dict, rag_retriever: SemanticRetriever):
        self.profile = character_profile
        self.retriever = rag_retriever
    
    def build_system_prompt(self) -> str:
        """每个 Agent 重写，定义自己的角色设定"""
        raise NotImplementedError
    
    def run(self, context: dict) -> dict:
        """接收上下文，返回本 Agent 的输出"""
        raise NotImplementedError
    
    def _call_llm(self, user_msg: str) -> str:
        system = self.build_system_prompt()
        return llm_client.chat(system, user_msg)
```

**五个 Agent 的协作顺序**：

```
1. DirectorAgent（导演）
   输入：剧目名称、参与角色列表
   输出：总体风格、主题、幕次规划
   
2. ScreenwriterAgent（编剧）
   输入：导演的总体规划
   输出：分幕大纲（每幕情节、角色关系、冲突点）
   
3. ActorAgent（演员，每个角色各一个）
   输入：编剧大纲 + RAG检索到的该角色历史表演片段
   输出：该角色在每幕的具体台词、唱腔
   关键：RAG 注入让台词风格忠于该角色历史表演
   
4. SceneSettingAgent（场景设定）
   输入：编剧大纲 + 演员台词
   输出：每幕舞台布景、灯光设计、音效 JSON
   
5. CostumeDesignerAgent（服装设计）
   输入：角色档案（Profile）+ 剧情背景
   输出：每个角色的服装、头饰、脸谱、道具 JSON
```

### 3.4 RAG 上下文注入（核心创新点）

**ActorAgent 的 RAG 增强**是本项目最关键的技术点：

```python
# context_builder.py
class ContextBuilder:
    def build_actor_context(self, character: str, scene_desc: str) -> str:
        # 1. 根据当前场景检索历史表演片段
        historical_refs = self.retriever.retrieve(
            query=scene_desc,
            character=character,
            top_k=5
        )
        
        # 2. 按类型分组（唱腔/念白/动作）
        singing_refs = [r for r in historical_refs if r['chunk_type'] == 'singing']
        dialogue_refs = [r for r in historical_refs if r['chunk_type'] == 'dialogue']
        
        # 3. 构建注入上下文
        context = f"""
        【历史唱腔参考】（{character} 在类似场景下的历史演唱）
        {format_references(singing_refs)}
        
        【历史念白参考】
        {format_references(dialogue_refs)}
        
        请模仿以上历史风格，为当前场景创作台词，保持角色一致性。
        """
        return context
```

**为什么这样设计有效**：
- 注入的参考来自真实历史演出，不是 LLM 的"想象"
- 同一角色的演出风格有延续性，检索到的片段提供了风格锚点
- LLM 的"模仿"能力 + 具体参考 = 既有创新又保持风格一致

## 7. 面试高频问答

### Q1: 介绍一下你的京剧多智能体项目

**参考回答**：

这个项目叫 Beyond Dialogue，是一个面向京剧表演的多智能体剧本生成系统。

核心痛点是：让 AI 生成京剧剧本，直接用通用 LLM 有三个问题——没有角色个性、不懂京剧规范、没有历史数据支撑。

我的方案是三层架构：首先建立数据管线，把 56 部历史剧本（PDF/TXT）经过 LLM 结构化提取，构建角色知识库；然后用 FAISS 建立语义向量索引，支持按剧情语义检索历史表演片段；最后五个专业 Agent 协作生成剧本——导演定方向、编剧写大纲、演员生成台词（RAG增强）、服装和场景设计完善细节。

关键创新是 RAG 增强的 ActorAgent：写每段台词之前，先检索"该角色历史上在类似场景下唱了什么"，把检索结果注入提示词，让 LLM 在风格上"模仿"历史演出而不是凭空创作。

输出是全要素的：不只有剧本文本，还有场景设定 JSON、装扮设计 JSON、评估报告，以及可直接对接视频生成 API 的 Prompt 文件。

---

### Q2: RAG 系统具体是怎么工作的？

**参考回答**：

RAG 分两个阶段：离线索引构建和在线检索增强。

**离线阶段**（一次性，提前做好）：
把 56 部增强剧本按段落切片（chunk_size=512，overlap=64），每个切片调用 Embedding API 编码成 1536 维向量，存入 FAISS 的 IndexFlatL2 索引。同时保存每个切片的元数据（所属角色、剧目、类型）到 documents.json。

**在线阶段**（生成时实时）：
ActorAgent 写某段台词时，把当前场景描述向量化，在 FAISS 里做 K 近邻搜索，然后用元数据过滤——只保留目标角色的片段，避免把别的角色的风格混进来。把 top-5 的历史片段拼接进提示词，让 LLM 参考历史风格生成。

**效果**：相比不用 RAG，生成的台词在唱腔格式、念白风格上与该角色历史演出的一致性明显更高。

---

### Q3: 为什么选择 FAISS 而不是其他向量数据库？

**参考回答**：

这个规模不需要专业向量数据库。

我的数据量：56 部剧本，每部约 20 个切片，总计约 1000 条向量。FAISS 的 IndexFlatL2 在这个量级下毫秒级响应，完全满足需求。

FAISS 的优势：
1. **本地运行**：不需要服务端，部署简单，无额外依赖
2. **序列化方便**：`faiss.write_index()` / `faiss.read_index()` 直接持久化
3. **轻量**：只有一个 `.index` 文件，配合 `documents.json` 元数据即可

如果数据量扩展到百万级，会考虑 Chroma 或 Weaviate（有元数据过滤支持），或者 Milvus（分布式）。

### Q4: 多智能体之间如何传递信息？有没有一致性问题？

**参考回答**：

信息传递通过结构化 JSON 上下文，由 `DialogueManager` 统一管理。每个 Agent 输出后，结果存入共享的 `session_context` 字典，下游 Agent 读取时直接从这个字典取。

一致性问题的解法：
1. **单向依赖链**：导演 → 编剧 → 演员 → 场景/服装，不存在循环依赖
2. **强制上下文传递**：后置 Agent 的提示词里强制包含前置 Agent 的关键输出（如演员生成台词时必须包含编剧大纲的幕次结构）
3. **格式约束**：用 JSON Schema 约束每个 Agent 的输出格式，下游 Agent 能可靠解析
4. **评估兜底**：最终评估报告会检查跨 Agent 的一致性（台词与场景是否匹配、服装与角色身份是否符合）

### Q5: 这个项目和普通的 LLM 剧本生成有什么本质区别？

**参考回答**：

有三个本质区别：

**1. 数据驱动 vs 想象驱动**：本项目基于真实历史剧本数据，RAG 检索确保生成有据可查。普通生成完全依赖 LLM 的训练记忆，无法保证角色风格一致性。

**2. 专业化分工 vs 单一模型**：五个 Agent 各有专业系统提示，CostumeDesignerAgent 懂京剧行头规范，SceneSettingAgent 懂舞台布景规制。普通方案用一个 Prompt 什么都要，往往哪方面都不专业。

**3. 全要素输出 vs 纯文本**：除了剧本，还输出结构化的场景设定 JSON 和装扮设计 JSON，可以直接对接下游数字化系统（视频生成、3D 渲染等）。普通方案只输出文字，无法进入数字生产流程。
