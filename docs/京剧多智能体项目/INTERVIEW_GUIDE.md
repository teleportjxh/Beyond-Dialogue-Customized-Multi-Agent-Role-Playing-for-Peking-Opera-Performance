# 京剧多智能体项目（Beyond Dialogue）面试指南

---

## 面试口语化介绍

我做了一个叫 Beyond Dialogue 的项目，全名是"面向京剧表演的定制化多智能体角色扮演系统"。

核心想法是这样的：现在 AI 生成文本已经很厉害了，但直接让 ChatGPT 写京剧剧本有一个根本问题——它生成的内容没有"角色个性"。比如让它写诸葛亮的唱词，它确实能写出来，但写出来的诸葛亮和孙悟空可能没什么区别，也不懂西皮二黄这些京剧专业规范。

我的解法是三层架构：

第一层，**数据管线**：收集了 56 部真实京剧历史剧本（PDF 和 TXT 格式），用 LLM 把它们结构化提取成 JSON，区分唱腔类型、板式、念白风格、身段动作，构建角色知识库。

第二层，**RAG 语义检索**：用 text-embedding-ada-002 把所有剧本片段向量化，存入 FAISS 索引。生成剧本时，根据当前场景语义检索"这个角色历史上在类似情境下唱了什么"，把检索结果注入提示词——这样 LLM 不是凭空想象，而是在参考真实历史演出风格。

第三层，**五智能体协作**：仿照真实京剧创作团队，设计了五个专业 Agent——导演、编剧、演员、服装设计师、场景设定师——按顺序协作，每个 Agent 的提示词高度专业化，输出互相依赖和参考。

最终输出是全要素的：一个剧目会同时生成剧本文本、场景设定 JSON、装扮设计 JSON 和质量评估报告。还有一个下游的视频生成子系统，能把剧本转成 Sora 等视频模型可用的提示词。

## 高频面试问答

### Q1: 为什么要做这个项目？解决了什么真实问题？

**参考回答**：

京剧是非物质文化遗产，有严格的表演规范和深厚的角色体系，但目前 AI 生成领域对这类**垂直领域专业内容**的支持非常弱。

通用 LLM 的问题在于：
- 它不知道"诸葛亮是老生行当，演唱以西皮慢板为主"
- 它不知道"念白有韵白和散白之分，不能乱用"
- 它更不知道"特定演员版本的诸葛亮有哪些标志性唱腔"

我这个项目本质上是在做**领域知识注入**：通过历史数据 + RAG 把专业知识"喂"给通用 LLM，让它在特定领域表现出专业水准，同时保留 LLM 的创作灵活性。

这个思路不只适用于京剧，任何需要"个性化 + 专业规范"的内容生成场景都可以复用：法律文书起草、医疗问诊记录、特定风格的文案写作等。

---

### Q2: RAG 和直接把所有剧本塞进上下文有什么区别？

**参考回答**：

这是个很好的问题。理论上，如果上下文窗口足够大，可以把所有 56 部剧本全放进去。但有几个问题：

**1. Token 成本**：56 部剧本大约有几十万字，全放进上下文每次调用费用极高。

**2. "迷失在中间"现象（Lost in the Middle）**：研究表明，LLM 处理超长上下文时，对中间部分的信息利用率显著下降。把所有剧本塞进去不代表它能有效利用所有内容。

**3. 精准性**：RAG 检索的是**与当前场景最相关的片段**，注入的是"诸葛亮在危机场景下的表现"，比无差别地给全部内容更有针对性。

**4. 灵活性**：新增一部剧本只需要更新索引，而不用修改提示词结构。

所以 RAG 在成本、精准度、可扩展性上都优于"全文塞入"。

---

### Q3: 你怎么保证五个 Agent 的输出是一致的？

**参考回答**：

一致性是多智能体系统的核心挑战。我用了三个机制：

**1. 单向依赖链**：导演 → 编剧 → 演员 → 场景/服装，严格单向，不存在循环依赖。后置 Agent 必须读取前置 Agent 的完整输出。

**2. 强制上下文注入**：比如演员生成台词时，提示词里强制包含编剧的幕次大纲——"第一幕：曹操宴请刘备，刘备表现谦逊"。演员不能自己"发明"一个第三幕。

**3. 结构化 JSON 约束**：每个 Agent 的输出都有固定的 JSON Schema，`DialogueManager` 在传递上下文前会先验证格式，避免因格式错误导致下游 Agent 解析失败。

如果出现明显的不一致，最终的评估报告会指出（比如"台词中出现了冬天的场景，但场景设定是夏天"），可以触发重生成。

### Q4: 你的数据是从哪里来的？数据处理过程中遇到了什么问题？

**参考回答**：

数据来源是公开的京剧剧本资源（PDF 格式）和文字版本，涵盖诸葛亮、孙悟空、赵匡胤三个角色，共 56 部剧目。

数据处理遇到的最大问题是**格式不统一**：
- 有的 PDF 是双栏排版，提取出来的文字顺序是乱的
- 唱词和念白的标记方式各不相同，有的用括号，有的用行距区分，有的完全没有标记
- 部分剧本有繁简体混用、异体字等问题

解决方案是用 LLM 做结构化提取，而不是依赖规则。给 LLM 一个目标 JSON Schema，让它理解语义后提取，比写正则表达式更鲁棒。

失败的情况（LLM 输出不合格 JSON、内容截断等）会记录到 `error_json/` 目录，统一批量重试，而不是中断整个流程。

### Q5: FAISS 的 IndexFlatL2 和 IndexIVFFlat 有什么区别？你为什么选前者？

**参考回答**：

- **IndexFlatL2**：精确搜索，对所有向量计算距离，结果100%准确，速度是 O(n)
- **IndexIVFFlat**：近似搜索，先用倒排索引缩小搜索范围，速度更快但可能遗漏最近邻

我选 IndexFlatL2 的原因：

**数据量决定选型**。我的索引里总共约 1000 条向量（56 部剧本 × 约 20 切片）。在这个量级下，IndexFlatL2 毫秒级响应，完全没有性能瓶颈。IndexIVFFlat 需要预先训练聚类中心（`nlist` 个聚类），在 1000 条数据上反而会因训练数据不足导致聚类效果差。

**一般经验**：
- < 10 万向量：IndexFlatL2（精确，够快）
- 10 万 ~ 100 万：IndexIVFFlat（近似搜索，可接受误差）
- > 100 万：IndexIVFPQ（量化压缩，节省内存）

### Q6: 这个项目和 LangChain 那套框架有什么区别？为什么不用现成框架？

**参考回答**：

LangChain、AutoGen、CrewAI 这些框架确实提供了 Agent 编排和 RAG 的现成实现。但我选择自己实现有几个原因：

**1. 学习目的**：从零实现让我真正理解了 RAG 的每个环节——切片策略怎么选、FAISS 索引怎么建、元数据过滤怎么做。用框架封装后这些细节都隐藏了。

**2. 高度定制化**：京剧场景有领域专业性，比如唱腔类型过滤（只检索同类型的唱腔片段）、角色严格隔离（不同角色的风格不能互相污染）这些逻辑在通用框架里很难直接实现，反而要绕过框架的抽象层。

**3. 依赖可控**：LangChain 版本迭代非常快，API 经常变化，引入重依赖后维护成本高。自己实现的代码完全可控。

如果是生产项目，会考虑用 LangGraph 做工作流编排（它的有向图模型和我的单向依赖链很匹配），但核心的 RAG 检索逻辑仍然自己控制。

### Q7: 如何评估生成剧本的质量？

**参考回答**：

用 **LLM-as-Judge** 方式，让一个独立的评估 Agent 对生成剧本打分，评估维度包括：

| 维度 | 权重 | 说明 |
|------|------|------|
| 角色一致性 | 30% | 台词风格是否符合该角色的历史表演特征 |
| 京剧规范性 | 25% | 唱腔板式标注是否正确、念白类型是否合适 |
| 情节合理性 | 25% | 故事逻辑是否通顺，与大纲是否匹配 |
| 跨 Agent 一致性 | 20% | 台词、场景、服装之间是否相互匹配 |

评估结果存为 JSON，支持自动触发重生成（`scripts/evaluation/regenerate_script.py`）。

评估的局限性：LLM 评 LLM，可能存在偏差。理想情况是邀请京剧专业人士做人工评估，但这个项目目前以技术验证为主。

### Q8: 项目的下一步如何扩展？

**参考回答**：

有几个自然的扩展方向：

**1. 扩充角色库**：目前只有三个角色（诸葛亮/孙悟空/赵匡胤），可以扩展到更多京剧人物，建立更大的知识库。

**2. 多角色同场戏**：目前一次生成以单一角色为主，扩展支持多角色对戏（如诸葛亮 vs 周瑜的交锋），需要在 ActorAgent 之间增加"对话协商"机制。

**3. 微调专用模型**：收集足够多的京剧数据后，可以用 LoRA 微调一个专门的京剧生成模型，减少对 RAG 的依赖，降低推理成本。

**4. 端到端视频生成**：目前视频生成子系统（`vedio_generation/`）已经能生成 Sora 的 Prompt，下一步是接通视频生成 API，实现从"剧目名称"到"数字京剧视频"的完整闭环。
